# 코드 아키텍처 가이드 — 시스템별 구조와 설계 근거

> 각 절 = [구조 → 왜 → 동작 → 예상 질문].

## 0. 30초 요약 (엘리베이터)

"핀볼 디펜스 1스테이지 모작입니다. **순수 C# 코어**(규칙·계산)와 **Unity 레이어**(물리·표시)를 분리해서 코어는 EditMode 테스트 42개로 커버했고, 게임 수치는 전부 **CSV 데이터 주도**입니다. 매니저들은 싱글톤 대신 **GameManager가 프로퍼티로 주입**하고 서로 이벤트로 통신합니다. 특징적인 설계는 세 가지 — 회수된 볼이 다음 발사가 되는 **볼 인벤토리**, 모든 데미지에 출처를 태깅해 스킬별 통계를 만드는 **소스 태그 파이프라인**, 시간이 아니라 '한 칸 거리'로 박자를 정의한 **행 컨베이어 스폰**입니다."

## 1. 전체 지도

```
GameManager (상태 머신: Ready→GamePlay⇄SkillSelection/GameStop→GameClear/GameOver)
 │  Awake에서 "Manager" 태그 오브젝트들을 RegisterManager → 프로퍼티 할당 → Initialize 루프
 │  (프로퍼티 할당이 전부 끝난 뒤 Initialize라서, Initialize 안에서 다른 매니저 읽기가 안전)
 ├ FieldManager    격자·벽의 SSOT — "배경 그림 bounds 안 비율"에서 파생 (§6)
 ├ BallManager     비매니저(볼/슈터)가 접근하는 유일한 창구
 ├ MonsterManager  스폰/필드 점유/이벤트 중계 (몬스터 개체는 매니저를 모름)
 ├ WaveManager     행 컨베이어 (§5)
 ├ SkillManager    스킬 규칙 조정자 — 순수 코어를 소유하고 이벤트와 잇는다 (§4)
 ├ PlayerManager   플레이어 HP
 ├ StatsManager    수치 집계 단일 창구 (§3)
 └ GameUIManager   UIElement들에게 (gameManager, 자신)을 주입 (§7)

프리팹 인스턴스(Ball/Monster)는 매니저를 모른다 — 값 주입 + 이벤트 발행만.
```

**왜 싱글톤이 아닌가?** 싱글톤은 의존성이 코드 어디서나 숨어들어 추적이 어려워짐(Unity 공식 e-book도 지적). 프로퍼티 주입은 "누가 무엇을 아는가"가 GameManager 한 곳에 명시된다. 대신 매니저 추가 시 3종 세트(프로퍼티+Register+태그)를 지켜야 하는 비용을 감수.

## 2. 볼 인벤토리 — "회수가 발사를 만든다"

```
BallInventory (순수 C#, Queue<SkillId?>)
  Add(스킬)  TryTakeNext(out 스킬)  Return(스킬)
BallShooter: 타이머마다 TryGetNextLoadout 성공 시만 발사 / 볼이 바닥에 튕겨 슈터로 돌아오면 Return
```

- **왜 큐인가**: 원작 규칙이 "먼저 회수된 볼부터 재발사" — 큐가 자료구조 그 자체. 쿨다운 개념이 없고 **회수가 곧 발사 자원**.
- 노멀볼 = null 원소 (스킬 없는 볼). 시작 5발 (원작 재확인 정정). "노멀 볼" 카드를 뽑으면 개수+레벨 성장.
- 레벨/데미지는 **발사 시점에 조회** — 비행 중 레벨업이 다음 발사부터 반영 (스냅샷 문제 회피).
- 드래프트 후보가 3장 미만이면 노멀 볼 카드로 채움 → 선택창이 절대 빈 화면이 안 됨 (구조적 해결 — 예외처리가 아니라 규칙으로).

**예상 질문** "볼이 사라지면(버그로) 어떻게 되나?" → 발사 자원이 줄어든 채 지속. 안전망으로 볼 탈출/정지 감시가 회수 모드로 강제 전환.

## 3. 데미지 파이프라인 + 소스 태그 (오늘의 핵심 자랑거리)

```
가해자(6곳)가 자기 정체를 태그해서 때린다:
  볼 직격 → TakeDamage(dmg, crit, ball.ActiveSkill)   ← null이면 노멀
  화상 틱 → (…, FireBall)   레이저 행 → (…, LaserBall)
  파편 → (…, ClusterBall)   성냥 폭발 → (…, LastMatch)

Monster.TakeDamage: HP 적용만. OnDamaged(자신, dmg, crit, source) 발행
 → MonsterManager: 팝업 표시 + OnDamageDealt(source, dmg) 재방송
 → StatsManager: CombatStats(순수)로 누적
 → CombatInfoPanel: 열 때 스냅샷 표시 (누적·최대 대비 비율·DPS)
```

- **계산과 적용의 분리**: 데미지 공식(양철심장/거울/빙결/치명타 조합)은 순수 `DamageCalculator` — 테스트로 조합 검증. Monster는 결과만 받음.
- **왜 StatsManager 신설?** 집계를 SkillManager에 넣으면 SRP 붕괴. "이벤트로 흘러오는 수치는 한 창구로"(레퍼런스의 InGameCountManager 패턴). UI는 이 창구 하나만 봄.
- **null 키 사건**: 노멀볼 소스가 null인데 Dictionary는 null 키 금지 → 첫 타격 크래시. 딕셔너리 경계에서 null→-1 예약 키 번역 함수 하나로 해결 — 호출부 6곳 무변경. "구현 제약은 경계에서 흡수"의 실례.
- DPS 분모 = 스케일 경과 시간(Time.time) — timeScale 0인 선택/퍼즈 시간이 자동 제외.

## 4. 스킬 시스템 — 데이터(CSV) × 행동(클래스) × 규칙(순수)

```
SkillTable.csv → 엄격 파서 → SkillDef (수치 a/b, 이름, 설명, 아이콘 경로 전부 데이터)
PlayerSkills   보유/레벨 (Dictionary<SkillId,int>)     ← "~별 관리 = Dictionary"
SkillDraft     3택지 규칙: 중복없음·신규Lv1·보유+1·만렙 제외·모자라면 노멀 카드
IOnHitEffect   스킬 행동 1클래스 1파일 (Fire/Ice/Laser/Cluster) — 디스패치는 Dictionary
```

- **OCP**: 새 스킬 = CSV 한 줄 + 효과 클래스 한 파일. 기존 파일 수정 없음.
- **왜 파서가 엄격한가**: 잘못된 데이터가 조용히 0으로 들어가면 밸런스 버그로 위장함. 즉시 예외가 디버깅 비용이 압도적으로 싸다 (실제로 오염 행을 잡은 이력 2회 — 스킬 표, 스테이지 패턴).
- 패시브는 계산기 입력으로만 존재 (양철심장=노멀 보너스, 거울=벽튕김 비례, 단검=면 조건 치명타 1회).

## 5. 행 컨베이어 + 멀티셀 (웨이브의 정체)

```
StagePattern.csv: 행당 9글자 (.빈칸 / 1·2타입 / D사슴앵커 / B돌벌레앵커 / -점유)
StagePattern.Parse: 멀티셀 점유 정합 검증 (D 위 칸=다음 행 '-', B 오른쪽='-')
WaveManager: 시작 5행 일괄 → "앞 행이 한 칸 내려온 시간"(CellHeight÷moveSpeed)마다 다음 행
클리어 = 행 소진 AND 필드 전멸
```

- **핵심 결정 — 박자를 거리로 정의**: 초로 정하면 속도 튜닝 때마다 줄 간격이 어긋남. 거리로 정하면 어떤 속도에서도 줄들이 정확히 한 칸 간격 = 세로 1×2 몬스터의 "윗줄 자리 선점"이 성립하는 전제.
- 빈 행 = 패턴의 일부 (원작의 그룹 간격). 결정적(Random 없음) — 매판 동일.
- 멀티셀: 앵커 칸 기준 점유 중심에 배치, **블록 스프라이트 자체가 1×2/2×1 크기**라 균등 스케일 유지(아트 왜곡 0), 콜라이더만 확장. 진행도 분모는 유닛 수(멀티셀=1).
- WaitForSeconds(스케일)라 선택/퍼즈/결과 중 컨베이어도 정지 — 상태 관리 공짜.

## 6. 필드 지오메트리 — "그림이 SSOT"

```
FieldManager.Initialize: 배경을 화면에 펴기 → 그 bounds의 (13%~93%, 22%~80%) = 필드
LeftWall/…/CellToWorld(행,열) 전부 여기서 파생 — 소비자는 좌표 계산을 모름
```

- **역사**: 화면 뷰포트 %로 잡았다가 화면비 변화에서 그림과 필드가 어긋남(실기기 버그) → 기준을 그림 하나로 통일. "두 시스템이 어긋나면 기준이 두 개는 아닌지 의심."
- 벽 콜라이더는 그림보다 wallPadding(0.15)만큼 바깥 — 조준선 반사점 표시도 같은 값으로 보정(짝 상수).
- 스폰은 판 상단 "한 행 아래"부터(원작 관찰) — 상단 완충 행이 볼-스폰 겹침 낑김을 원천 차단.

## 7. UI 체계 — 에디터-네이티브 + UIElement 주입

```
원칙: 배치/색/크기 = 씬(Inspector) / 코드 = 참조와 로직만
복잡 조립(행 7개짜리 패널 등) = InGameUIBuilder 에디터 메뉴 (1회 생성 도구, 빌드 제외)
UIElement 상속 = "GameUIManager가 주입·관리하는 단위" (팝업+상시 모두)
  gameManager/gameUIManager는 주입받음 — serialized 참조 0
  월드 오브젝트(DeathPose 등)만 예외: 창구 참조 1개 허용
게이지 = 전부 UGUI Slider(핸들 없음, interactable off) + MoveTowards 보간
```

- **왜 손 YAML 금지?** Slider/TMP를 손으로 직렬화하다 dangling 참조·씬 파서 파손 2연발 — "복잡 컴포넌트는 에디터 API로만"을 규약화.
- 몬스터 체력바만 SpriteRenderer — 개체 수십 개에 캔버스 슬라이더는 리빌드 비용(모바일 관례). "화면에 1개 = Slider, 다수 개체 = SpriteRenderer".

## 8. 성능 결정 3개

1. **전부 오브젝트 풀** (볼/몬스터/파편/팝업) + 데미지 팝업 50개 사전 워밍 — 대량 처치 프레임의 Instantiate+TMP 메시 생성 폭풍이 에디터 프리즈까지 유발했던 실사례의 해법. "스킵/상한 없이" 버티는 게 목표였음 (원작도 안 버림).
2. 물리 콜백 "도중" 콜라이더 제거 금지 — 볼 반사가 반쪽 계산되어 속도 0이 되는 버그 → 죽음의 물리적 제거는 LateUpdate로 지연 (논리적 죽음은 즉시).
3. Update 내 할당 회피 — 조준 점 풀, 궤적 리스트 재사용, HUD는 변경 감지 캐시로 문자열 할당 최소화.

## 8-1. HP 밸런스 확정 (2026-07-07)

플레이어 HP 300 / 몬스터 도달 1회 데미지 50 = **6대 흘리면 사망**.
근거: 기존 10(30대)은 사실상 무사망이라 긴장감 0 — "한두 마리는 괜찮지만 쌓이면 죽는다"가
원작류 디펜스의 압박감. 수치는 Inspector(MonsterManager.attackDamage) 노출이라 조정 비용 낮음.

## 9. 예상 질문 속사포

- **"왜 순수 C# 분리?"** 규칙을 엔진 없이 테스트(42개, 1초). 드래프트 규칙·데미지 조합·회수 순서는 Play로 재현이 사실상 불가능.
- **"이벤트 해제는?"** 구독/해제를 쌍으로 — 핸들러 내부(1회성), 매니저 Clear(씬 리로드), OnDestroy 중 한 곳에 명시. 설계 검사 항목.
- **"재시작 처리?"** timeScale 복원 → 씬 리로드. 매니저 Clear가 구독·풀 정리. 상태 진입 액션(AddGameStateEnterAction)으로 부수효과 등록.
- **"멀티셀 충돌 판정?"** 콜라이더 크기만 점유 칸으로 — 레이저 "같은 행"은 Y밴드 쿼리라 자동 대응.
- **"제일 어려웠던 버그?"** §8-2 볼 정지 (물리 콜백 타이밍) — 증상은 랜덤 볼 멈춤, 원인은 사망 처리 중 콜라이더 제거. 근본 수정+감시 안전망 이중 구조로 해결.
- **"AI 활용?"** 페어 프로그래머로 — 단 설계 승인 게이트/원작 관찰 우선/전 코드 워크스루를 규약으로 강제 (README에 명시). 이 가이드를 내가 설명할 수 있는 것이 그 증거.
