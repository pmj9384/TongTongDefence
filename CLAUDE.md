# TongTongDefence — 개발 사이클 (모든 세션 필수 준수)

PurpleCow 채용과제. **유저가 모든 코드를 면접에서 설명할 수 있어야 하므로, 유저의 이해가 결과물의 일부다.**
혼자 달리지 말 것 — 아래 사이클 밖에서 코드 파일을 수정하지 않는다.

## 사이클 (기능/리팩토링/버그픽스 공통)

1. **태스크 등록** — TaskCreate로 등록하고 내용 설명 (무엇을/왜/어느 파일/완료 기준)
2. **설계 검사** — 새 시스템·구조 변경이면 design-check 스킬 → 플랜 문서(Obsidian 구현플랜). 작은 변경이면 "변경 목록"을 채팅에 제시
3. **승인 게이트** — 유저의 명시적 "ㄱ / 시작 / 해" 후에만 첫 Edit.
   - **질문형 응답("~아닌가?", "~하면 되지 않나?")은 승인이 아니다 — 답만 하고 멈춘다**
   - 짧은 동의("그렇네", "ㅇㅋ")도 진단에 대한 동의일 수 있음 — 착수 직전 "이렇게 바꾼다" 목록 재확인
4. **구현** — 승인된 범위만. 범위 밖 발견사항은 코드 건드리지 말고 새 태스크로 등록
5. **검증** — 컴파일은 내가 선확인(콘솔/recompile), 동작은 **유저 Play Mode** (체크리스트 제시)
6. **커밋** — 검증 통과 즉시, 논리 단위로 쪼개서, 메시지에 why 포함
7. **기록** — 플랜 문서 체크박스·메모리 갱신

## 규모별 축약

- **값 튜닝(숫자 1~2개)**: "X를 Y로 바꿈, ㄱ?" 한 줄 승인으로 1~3단계 대체
- **버그**: 진단 → 원인+수정안 제시 → 승인 → 구현 (진단했다고 바로 고치지 않는다)
- **새 시스템**: 원작 관찰/기획서 대조(0-A)가 설계보다 먼저

## 중단 규칙

- 방침이 **2회 이상 뒤집히면 즉시 멈추고** "확정 스펙 표"로 정리해 승인받은 뒤 일괄 반영
- 유저가 이해 못 한 채 넘어가는 신호가 보이면 멈추고 설명부터

## 개발 모드 (스킬로 운영 — 2026-07-06 확립)

- **기본 = `/dev-cycle`** (동기): 위 사이클의 운영 체크리스트 버전. 새 설계/로직은 전부 이 모드
- **예외 = `/auto-mode`** (배치): 유저 명시 발동 + 사전 승인 설계만. **복귀 게이트 = 검증 → 워크스루 → [가정] 결정 → 머지** — "승인 없이 만들 수는 있어도 이해 없이 머지할 수는 없다"
- 구현 후 전수 리뷰는 `/refactor-check` (발견만, 수정은 승인 후 별도 태스크)

## 레퍼런스 스킬 (이름만 알지 말고 실제 호출)

- 플랜 문서 작성 시 `superpowers:writing-plans`, 완료 선언 전 `superpowers:verification-before-completion`,
  구현 후 리뷰 시 `superpowers:requesting-code-review` — 해당 시점에 반드시 Skill 툴로 호출해 체크리스트를 탄다.
- 이 사이클은 superpowers 워크플로우 + GitHub Flow의 프로젝트 튜닝 인스턴스다.

## 프로젝트 관례 (요약 — 상세는 design-check 스킬)

- 매니저: InGameManager 상속, 태그 "Manager", GameManager 프로퍼티+RegisterManager 등록 3종 세트
- 매니저 간 참조는 `GameManager.{Manager}` 경유. 프리팹 인스턴스(Ball/Monster)는 매니저를 모름 — 값만 전달
- 오브젝트 생성은 ObjectPool. FindObjectOfType/GameObject.Find 금지
- 필드 지오메트리는 FieldManager가 SSOT — 위치는 CellToWorld(행,열)로 지목
- 커밋 워크플로우: 구현 → 유저 Play 확인 → 즉시 커밋 → 다음 세션 리뷰
- **브랜치 (git-flow 축약)**: feature/* → **develop** 머지 (Task/기능 단위 분기, 한 브랜치 몰빵 금지).
  **master는 빌드용** — develop이 빌드 검증(APK)된 시점에만 develop → master 머지
