# 통통 디펜스 (TongTong Defence)

> **통통 디펜스: 핀볼 마스터** 1스테이지 모작 — PurpleCow 채용 과제
> Unity 6000.3.10 · Android(세로) · 개발 기간 6일 (마지막 날은 검증·문서화)

<!-- 플레이 GIF: 영상 촬영 후 삽입 (조준→반사→스킬→레벨업 15초 컷) -->

몬스터를 향해 볼을 쏘아 튕기며 싸우는 핀볼 디펜스입니다. 원작 1스테이지의 핵심 루프 — 발사, 물리 반사 전투, 레벨업 3택지, 스킬 성장, 웨이브, 결과 — 를 재현했습니다.

---

## 1. 실행 방법

### APK (권장)
`TongTongDefence.apk`를 Android 기기에 설치하면 됩니다. IL2CPP · ARM64/ARMv7 빌드라 64비트 전용 최신 기기에서도 설치됩니다. (Min SDK 25, 세로 고정)

### Unity 에디터
압축 해제한 폴더를 Unity Hub에 추가하고 **Unity 6000.3.10**으로 열어 `Assets/Scenes/SampleScene.unity`를 Play 하면 됩니다. (첫 열기는 Library 재생성으로 수 분 걸립니다)

### 조작
드래그(터치)로 조준하면 점선 경로와 끝점 레티클로 반사 궤적이 미리 보입니다. 발사는 자동 — 보유한 볼이 회수되는 대로 다시 나갑니다. 우상단 ⏸로 일시정지, 일시정지 안에서 전투 정보(볼별 통계)를 볼 수 있습니다.

## 2. 주요 구현 내용

### 필수 요건 매핑

| 요건 | 구현 |
|---|---|
| 핀볼 로직 | PhysicsMaterial2D(탄성 1) 물리 반사 + 충돌마다 속력 재정규화(드리프트 방지) + 수평 잠금 방지(수직 성분 최소 보장) |
| 웨이브 20개 | 행 컨베이어 — 전투 행 20개 + 그룹 간격 행을 `StagePattern.csv`로 정의, 매판 동일(결정적). 클리어 = 행 소진 + 전멸 |
| 로그라이크 3택지 | 레벨업(5/10/15…킬) 시 정지 + 카드 3장 — 중복 없음 · 신규 Lv1 · 보유 +1레벨 · 만렙 제외 · 후보 부족 시 노멀 볼 카드로 채워 빈 선택창 원천 차단 |
| 스킬 10종 × 3레벨 | 액티브 볼 5(파이어/아이스/레이저/고스트/클러스터) + 패시브 5(양철심장/마법거울/단검 2종/성냥) — 수치는 전부 `SkillTable.csv` |
| 결과 + 재시작 | Stage Clear(남은 체력 %) / Fail(진행도 %) 팝업 + 1스테이지 재시작 (timeScale 복원 → 씬 리로드, 매니저 Clear로 구독·풀 정리) |

### 원작 관찰을 구현의 기준으로 삼았습니다

기획서에 없는 부분은 추측 대신 원작 스토어 영상과 실플레이를 관찰해 문서화하고, 그것을 구현의 기준으로 사용했습니다. 이 과정에서 처음 설계를 두 번 크게 뒤집었습니다.

**발사 모델**은 처음에 쿨다운 기반으로 만들었는데, 원작을 다시 보니 보유한 볼 하나하나가 개체였고 "먼저 회수된 볼부터 재발사"되고 있었습니다. 그래서 볼 인벤토리(큐)로 재설계했습니다 — 쿨다운이 아니라 **회수가 곧 발사 자원**이 되는 구조라, 큐라는 자료구조가 규칙 그 자체가 됩니다. (`BallInventory` — 순수 C# Queue, 시작 노멀 4발, 레벨·데미지는 발사 시점 조회라 비행 중 레벨업이 다음 발사부터 반영)

**웨이브**도 "전멸하면 다음 묶음 스폰"으로 만들었다가, 관찰 결과 원작은 시작에 5줄을 깔고 이후 한 줄씩 계속 이어지는 **행 컨베이어**였습니다. 다음 줄이 나오는 박자를 시간이 아니라 "앞 줄이 한 칸 내려온 거리"로 정의했는데, 이렇게 하면 하강 속도를 어떻게 튜닝해도 줄들이 격자 간격을 유지하고, 세로 2칸 몬스터가 윗줄 자리를 선점하는 것도 자연스럽게 성립합니다. (행 간격 = CellHeight ÷ 하강 속도로 자동 계산 — `WaveManager`)

### 규칙은 순수 C#으로 분리하고 테스트를 붙였습니다

데미지 공식, 카드 뽑기 규칙, 볼 회수 순서, 전투 통계, 행 패턴 검증 같은 "게임 규칙"은 엔진에 의존하지 않는 순수 C# 어셈블리(TongTong.Core)로 분리하고 EditMode 테스트 42개로 커버했습니다. 치명타 조합이나 파서의 오염 데이터 처리 같은 것들은 Play 모드로 재현하기가 사실상 불가능해서, 테스트가 유일하게 현실적인 검증 수단이라고 판단했습니다. 실제로 스킬 테이블의 오염된 행과 딕셔너리 null 키 크래시를 테스트와 엄격 파서가 잡아줬습니다.

```
TongTong.Core (asmdef, 엔진 무의존)
  SkillTableParser · DamageCalculator · SkillDraft · PlayerLevel
  BallInventory · CombatStats · StagePattern  ← 전부 EditMode 테스트 대상
```

### 매니저는 싱글톤 대신 주입으로 연결했습니다

싱글톤은 의존성이 코드 어디에나 숨어들어 추적이 어려워진다고 판단해, 중앙 GameManager(상태 머신)가 각 매니저를 프로퍼티로 들고 주입하는 방식을 썼습니다. "누가 무엇을 아는가"가 한 곳에 명시되고, 매니저 간 통신은 C# 이벤트로 — 볼·몬스터 같은 프리팹 인스턴스는 매니저의 존재를 모른 채 값을 받고 이벤트만 발행합니다.

이 구조가 가장 잘 드러난 곳이 **전투 정보 창**입니다. 원작 관찰 중 패시브(성냥 폭발)의 데미지까지 스킬별로 집계되는 걸 발견하고, 모든 데미지에 "누가 때렸는지" 소스 태그를 실어 보내도록 파이프라인을 만들었습니다. 볼 직격·화상 틱·레이저·파편·폭발 여섯 곳이 각자 자기 정체를 태깅하면, 집계 전담 매니저(StatsManager)가 스킬별로 모으고 UI는 그 창구 하나만 봅니다.

```
가해자(태그) → Monster.TakeDamage(dmg, crit, source)
  → OnDamaged 이벤트 → MonsterManager(데미지 텍스트 + 재방송)
  → StatsManager(CombatStats에 소스별 누적) → 전투 정보 창(누적·비율·DPS)
```

### 수치는 전부 데이터로 뺐습니다

스킬 수치·설명·아이콘은 SkillTable.csv, 스테이지 몬스터 배치는 StagePattern.csv로 — 코드를 열지 않고 밸런싱할 수 있습니다. 파서는 일부러 엄격하게 만들어 잘못된 데이터(멀티셀 점유 칸 불일치 등)를 즉시 예외로 던집니다. 조용히 0으로 들어간 값이 밸런스 버그로 위장하는 것보다 그 자리에서 죽는 게 디버깅 비용이 압도적으로 싸다고 생각합니다.

### 성능 관련 결정

볼·몬스터·파편·데미지 텍스트 전부 오브젝트 풀로 처리했고, 레이저+연쇄 폭발로 수십 마리가 한 프레임에 죽는 상황에서 데미지 텍스트 생성 폭풍이 프레임을 잡아먹는 문제를 겪은 뒤 풀 50개 사전 워밍을 추가했습니다. 또 물리 콜백 도중 죽은 몬스터의 콜라이더를 끄면 볼 반사가 반쪽 계산되어 속도가 죽는 버그가 있어서, 논리적 죽음은 즉시 처리하되 물리적 제거는 프레임 끝(LateUpdate)으로 미뤘습니다.

## 4. 가산점 구현

- **멀티셀 몬스터** — 사슴 1×2(세로) · 돌벌레 2×1(가로). 패턴 CSV에서 앵커 코드(D/B)와 점유 칸(`-`)으로 표현하고 파서가 정합을 검증합니다 (사슴 위 칸이 비어 있으면 로드 시점에 예외). 전용 블록 스프라이트(Block_1x2/2x1)로 비율 왜곡 없이 배치하고 콜라이더만 점유 크기로 확장했습니다
- **스킬 10종 전부 실동작** — 화상(스택형 도트) · 빙결(감속+받는 피해 증가) 상태이상, 레이저 행 일괄 피해, 고스트 관통, 클러스터 파편, 성냥 연쇄 폭발, 전·후면 조건 치명타
- **전투 정보 창** — 볼·스킬별 [아이콘 · ◆레벨 · 누적 데미지 · 최대 대비 비율 바 · DPS]. 성냥 폭발 같은 부가 피해도 소스별 귀속 — 원작 스크린샷을 대조해 확인한 사양입니다
- **EditMode 테스트 42개** — 순수 코어 규칙 전수
- **세이프 에어리어 대응** — 노치 기기에서 HUD가 가려지지 않습니다
- **에디터 도구** — UI 조립·프리팹 생성·원버튼 APK 빌드 메뉴 (Assets/Editor/)

## 5. AI 활용

Claude Code를 개발 전 과정에서 활용했습니다. 원칙은 역할 분담입니다 — **구조에 대한 판단과 설계 결정은 제가 내리고, 결정된 설계의 전개와 반복 작업을 AI가 맡습니다.**

이 분담이 말로 끝나지 않도록 프로젝트 규약(CLAUDE.md)으로 강제했습니다. 새 시스템은 SOLID·Unity 패턴 체크리스트로 설계 검사를 통과하고 제가 승인한 뒤에만 구현에 들어가고, AI의 추측성 판단은 전부 [가정] 태그로 격리해 원작 재관찰로 확정하거나 폐기합니다. 일괄 생성한 코드는 파일별 워크스루를 거친 뒤에만 병합했습니다 — 모든 코드를 제가 설명할 수 있는 상태가 이 과제의 요구라고 생각했기 때문입니다.

실제로 판단이 갈린 순간들이 있었습니다. 발사 로직을 쿨다운 기반으로 구현해둔 상태에서 원작을 다시 관찰해보니 "먼저 회수된 볼부터 재발사"라는 다른 규칙이 보였고, 만들어진 코드를 폐기하고 볼 인벤토리로 재설계하는 결정을 내렸습니다. 웨이브 모델도 같은 과정으로 한 번 뒤집었습니다. AI는 그럴듯한 구현을 빠르게 내놓지만, **그것이 맞는 구현인지는 관찰과 판단의 몫**이라는 걸 이 프로젝트에서 반복해서 확인했습니다.

이번 프로젝트에서의 구체적인 활용 방식은 이렇습니다.

- **MCP로 Unity 에디터에 직접 연결** — 씬 오브젝트 배치·컴포넌트 값 수정·메뉴 실행(APK 빌드 트리거까지)을 AI가 에디터 안에서 직접 처리하게 했습니다
- **에디터 도구 제작** — UI 조립·프리팹 생성·원버튼 빌드 메뉴를 AI와 함께 만들었습니다. 복잡한 컴포넌트를 손으로 직렬화하다 씬이 깨진 사고를 겪은 뒤 "복잡한 조립은 에디터 API 도구로만"을 규약으로 세우고, 그 규약을 도구로 만든 것입니다
- **실기기 버그 진단 파트너** — 다른 기기에서 볼이 상단에 갇혀 수평으로만 튕기는 버그 영상을 프레임 단위로 분석해 "수직 속도가 0으로 수렴하는 수평 잠금"으로 원인을 좁히고, 반사 시 최소 수직 성분을 보장하는 표준 해법으로 수정했습니다
- **개발 사이클 자동화** — 일지 작성이 다음날 작업 플랜의 입력이 되는 커스텀 스킬 사이클로, 세션이 바뀌어도 맥락이 끊기지 않게 했습니다

AI를 쓰는 것과 별개로, AI를 쓰는 방식 자체를 규약과 도구로 만들어두는 것이 품질과 속도를 같이 잡는 방법이라고 생각합니다.

## 6. 알려진 한계와 다음 단계

솔직하게 남겨둡니다.

- **사운드가 없습니다** — 기간 내 우선순위에서 게임 로직·원작 재현을 앞에 뒀습니다. 매니저 자리는 있어서 에셋과 호출부만 얹으면 됩니다
- **부활(광고 후 부활) 흐름은 스코프 아웃** — 원작 관찰로 존재는 확인했지만, 광고 SDK 연동이 과제 범위를 넘는다고 판단해 사망 → 결과 화면 직행으로 정리했습니다
- **밸런스는 1회 패스 수준** — 몬스터 HP 곡선(기본 30 + 행당 10)과 멀티셀 배수는 플레이 감각으로 잡은 추정치입니다. 수치가 전부 CSV/Inspector에 있어 조정 비용은 낮습니다
- **전투 정보의 DPS 분모는 관찰 기반 추정** — 원작이 정지 시간을 제외하는 것으로 보여 스케일 경과 시간을 썼습니다. 원작 수치와 1:1 대조는 못 했습니다
- 다음 단계로는 피격 연출(플레이어·몬스터 플래시), 데미지 텍스트 물리감, 사운드 순으로 폴리시를 계획하고 있습니다

## 7. 프로젝트 구조

```
Assets/
├ Scripts/
│  ├ Core/Managers/        GameManager(상태 머신) · ObjectPool · StatsManager 등
│  ├ Gameplay/             Ball · Monster · Shooter 계열과 매니저들
│  │  └ Skills/
│  │     ├ Core/           순수 C# 규칙 (TongTong.Core — 테스트 대상)
│  │     └ Effects/        온히트 효과 클래스 5개 (IOnHitEffect) — 고스트는 볼 거동, 패시브는 계산기 입력이라 여기 없음
│  └ UI/                   씬에서 조립하고 코드는 로직만 갖는 슬림 스크립트
├ Editor/                  1회 생성 도구 · 빌드 스크립트 (빌드 제외)
├ Resources/Tables/        SkillTable.csv · StagePattern.csv
└ Tests/EditMode/          코어 규칙 테스트 42개
```
